# 인프라 플래너 규칙서

테라 아카이브 인프라 플래너(`app/planner-engine.ts` + `app/planner.tsx` + `scripts/build-infra.py`)가
따르는 모든 도메인 규칙의 정본. 새 오퍼레이터가 나오거나 편성이 이상할 때 이 문서를 기준으로
파서·모델을 갱신한다. 데이터 갱신 절차 자체는 `scripts/README.md` 참고.

**규칙 계층 (2026-07-19)**: 절대룰·점수 모델은 `app/planner-engine.ts`(L0), 파싱된 게임 팩트는
`app/data/infra.json`(L1), **유동 규칙(추정 상수·토큰 카탈로그·파싱 교정·검증된 정배 픽스처)은
`app/data/rules.json`(L2)** — 계층 설계와 운영 흐름은 [PLANNER-RULES-DB.md](PLANNER-RULES-DB.md).
L2의 정본은 Supabase 원장이며 **/admin '플래너 규칙' 탭에서 편집·발행 →
`python3 scripts/build-rules.py` 베이크**로 갱신한다 (rules.json 손편집 금지).
엔진·rules.json·build-infra.py를 고치면 **커밋 전에 `node scripts/verify-plan.mjs`** (픽스처 회귀 검사,
엔진 리팩토링은 `--snapshot`→`--compare`로 무변화 증명)를 통과시킨다.

## 1. 기지 구성과 교대 정책

- 레이아웃은 **243 고정**: 무역소 2 · 제조소 4 · 발전소 3 + 제어 센터, 숙소 4, 응접실·가공소·사무실·훈련실.
- 제조소는 **순금 2 + 작전기록 2** 분할. KR 데이터 표기는 '금괴'가 아니라 **'순금'/'귀금속류'**.
- **A조가 풀파워 주력**이고 모든 시너지 세트는 A조에 집중한다. B조는 A조 컨디션 소진 시 투입되는
  회복 교대(12시간 2조). 3조 이상 교대는 쓰지 않는다.
- **같은 오퍼 A조·B조 동시 배치 금지 (기본 전제, 사용자 확정 2026-07)**: 근무를 이중으로 서면
  못 쉬고 24시간 돌게 되므로, 어떤 오퍼도 A·B 두 조의 **근무 방**에 동시에 들어가서는 안 된다.
  예외는 **사기를 소모하지 않는 숙소(휴식)·가공소(상시 슬롯)뿐** — 이 둘만 조 전환과 무관하게
  고정한다. 구현: ① 교차방 리페어 패스는 대체 후보에서 다른 조 근무 인원을 제외한다(동시 배치
  방지). ② 편성 후 불변식 패스가 A조를 남기고 B조 중복을 제거하되, **제거로 빈 자리는 벤치 최고
  요원으로 반드시 다시 채운다** — 근무 방(특히 제조소)을 비워 두지 않는다(사용자 확정 2026-07).
- **편성 후 전수 감사 — 조 단위 2단계 (사용자 확정 2026-07)**: 자동편성 시 `buildPlan`이 편성을 한 번
  완성한 뒤, **A조를 먼저 수렴할 때까지 전수검사로 풀파워로 완성**(전체 로스터에서 선발, 최대 8회 —
  변경이 없으면 조기 종료)하고, 그 다음 **B조를 "남은 오퍼만으로"** 같은 방식으로 검수한다.
  계산이 수 초 걸려도 전수조사 반복이 우선이다(사용자 확정 2026-07-19) — `optimize()`는 시너지
  세트 후보안도 **가능한 모든 조합(멱집합)**으로 만들어 총점 비교하고, UI는 진행 단계 안내를
  띄운다. A·B를 한 번에 섞어 보면 서로 자리를 뺏고 되돌리는 진동이 생긴다(사용자 확정). 각 회 순서:
  ① **방별 통째 검수** — **A조·B조 양쪽 모두**, 근무 방을 하나하나 보며 후보 세 팀을 만들어
  **총 생산력을 직접 비교**, 높은 쪽을 배치한다(한 자리 swap이 아니라 팀 전체 비교 — 국소최적
  방지. B조를 빼먹으면 "도로시 혼자" 방이 회복 교대에 남는다. 사용자 규칙 2026-07):
    ⓐ 현재 팀 그대로 / ⓑ 자유 풀(벤치, 반대 조 근무자 제외)로 방 전체 재편성한 최고 팀 /
    ⓒ **시너지 결집 팀** — perSkillTag 보유자(도로시: 라인테크류 1개당 +5%)를 축으로, **전체
    자유 풀을 한계 기여 그리디로 채우되 동률이면 계열 요원 우선**(+1e-4 타이브레이크). ⚠ 계열만
    먼저 채우는 조립은 금지 — [도로시+계열+계열]=95가 [도로시+계열+강범용]=100을 가리는 함정.
    계열 요원이 같은 조 **다른 방**에 있으면 그 방의 손실(차출 후 재충원해도 남는 점수 하락)을
    비용으로 계산해 **순증일 때만** 차출한다.
    → **총점 동률이면 ⓒ(시너지 결집)를 채택** — 같은 점수면 계열을 모아두는 게 정배(사용자 확정).
    도로시+라인테크(98%)가 범용 셋(91%)보다 세면 결집, 강범용이 더 세면 도로시 제외까지 자동.
    시드·예약 요원은 항상 유지.
  ② 조 동시 배치 제거(앞 조 근무자 잠금). ③ 빈 근무 방 재충원. ④ **체인 승격** — 벤치 또는
  같은 조 다른 방의 오퍼 S를 방 R의 멤버 M 자리에 넣고(손해 없음 이상), 밀려난 M을 다른 방의
  더 약한 자리/빈 슬롯으로 옮기는 2단 이동을 "관련 방 합산 순증"일 때 실행(타방 소스는 그 방을
  벤치로 재충원한 손실 g0 부담). 미즈키 30이 근무하고 트라고디아 35(작전기록 전용)가 벤치에
  남는 케이스, 미틈이 딴 무역소에 묶여 샤마르·테킬라 조합에 못 가는 케이스를 회수한다.
  안정되면 조기 종료. **개별 오퍼가 아니라 이 감사 규칙을 고쳐 편성 오류를 잡는다.**
  ⚠ 감사 전 단계는 **roomPartner 조건을 엄격 평가**(현재 배치 지도 `roomOf` 주입)해야 한다. 안 그러면
  낙관 평가로, 리페어가 뺀 "굼 없는 레토" 같은 데드웨이트를 감사가 도로 넣어버린다(롤백 전례 2026-07).
- **"~류" 스킬 계열 카탈로그 (사용자 확정 2026-07)**: 도로시("라인테크류 스킬 1개당 +5%")처럼 특정
  스킬 계열 수에 스케일하는 오퍼가 있어, `build-infra.py`가 계열(라인테크류·금속공예류·표준화류 등)을
  미리 정리한다 — 각 스킬에 `families`를 달고 최상위 `skillFamilies`에 태그→오퍼 목록을 emit.
  **계열 = 스킬명이 정확히 "<태그>" 또는 "<태그> α/β/γ"인 것만** (표준화류 = 표준화 α/β 등,
  사용자 확정 2026-07). 부분 문자열 매칭 금지 — '비표준화' 같은 이름의 오포함을 막는다.
  ⚠ 컨빅션 "작전기록**류**에 대한 생산력 +35%"의 '~류'는 **제품 분류**(순금/작전기록)지 스킬
  계열이 아니다 — perSkillTag 파서·계열 카탈로그와 무관. 스코어링·감사 폴백도 동일 정확 명칭 규칙.
  현행 카탈로그: 라인테크 7(사일런스·아스트젠·프틸롭시스·내스티·도로시·이격사일런스·이격아스트젠),
  금속공예 6, 표준화 13(야토·바닐라·…·하이모어·미즈키 — 전원 실제 '표준화 α/β' 보유자).
- **순금이 병목**: 무역소 효율이 오르면 순금 소비가 생산을 앞지르므로, 가장 강한 생산 팀을 순금
  2방에 먼저 앉히고 남는 효율을 작전기록으로 돌린다.
- **방 채우기 우선순위 (사용자 확정 2026-07)**: **제조소-순금 > 제조소-작전기록 > 무역소 >
  발전소 > 사무실 > 응접실** — 먼저 채우는 방이 최고 요원을 가져간다. 응접실은 최하위:
  쉐라처럼 응접실 스킬이 있어도 상위 방 세트(쉐라그 무역소)가 우선이다.
- **우선 생산 3모드 (사용자 확정 2026-07)**: 위 순서는 기본값(순금 우선)이고, **설정 행의
  라디오 버튼**으로 **작전기록 우선**(작전기록 방 2개를 먼저 채움)·**밸런스**(순금↔작전기록 교차)
  선택 가능. 방 순서만 바뀌고 나머지 규칙은 동일(`PRIORITY_KEYS`, 토큰 소비자 시드 배치 순서도
  모드를 따름). **라디오는 설정 저장뿐 — 실제 편성은 기존처럼 자동편성 버튼을 눌러 적용**
  (사용자 확정: 설정 변경이 현재 편성을 갈아엎으면 안 됨). 선택은 localStorage에 저장·복원.
- **훈련실 비움·가공소 니엔은 절대룰이 아니라 정배(점수 최적 결과)다 (사용자 정정 2026-07-21)**:
  둘 다 **점수 기반**으로 결정한다 — 하드 핀·강제 배치 금지. 가공소는 일반 방 채우기가
  최고 점수 가공소 오퍼를 넣는데, 표준 로스터에선 니엔(부산물 점수 최고)이 자연히 이겨 고정될
  뿐이다. 가공소에 더 높은 오퍼(예: 우르수스 특제 음료 토큰을 받는 Frost)가 있으면 그쪽이
  들어가고 니엔은 밀려날 수 있다(니엔을 훈련소 등에 돌리는 것도 가능 — 다만 니엔은 가공소
  스킬뿐이라 실제 대안 방은 없다). ⚠ 예전엔 "family pin"(토큰 생성자 진영의 가공소 동료를
  강제 배치)이 있었으나, 콜라보 토큰(아이룰루=작전팀 A4) family가 무관한 동진영 가공소 오퍼
  (레인저)를 니엔보다 먼저 앉히는 오발동을 내 **제거**했다. 훈련실은 "훈련 속도" 스킬이
  정상 운용 생산가치 0이라 채워도 점수가 안 올라 자연히 비는 것(하드 스킵은 스퍼리어스 배치
  방지용 모델링 선택)이지, 게임상 반드시 비워야 하는 건 아니다.
  가공소는 **상시 슬롯이라 A조 한 팀만 편성하고 B조 칸은 비워 둔다** — 사기 비소모 방이라
  교대 개념이 없고, 회복 교대에 가공 요원(혼 등)을 따로 채울 이유가 없다 (사용자 확정 2026-07-19).
  자동편성·빈 자리 채우기 모두 B조 가공소를 건드리지 않는다.
- **숙소는 항상 풀 인원(방당 5명) 기준**으로 계산하며, 조 전환과 무관하게 고정. 숙소 시간당 회복
  스킬로 배치를 최적화하지는 않는다 — 숙소 슬롯은 휴식 + 토큰 생성원 고정용.
- **응접실도 A/B 교대로 편성한다** (사용자 정정 2026-07: 같은 인원 24시간 금지 —
  구 모델의 '상시 편성' 특례 폐기).
- **정예화 단계 기본값 = 최대 정예화** (4성+ = 2정, 3성 = 1정, 1~2성은 승급 없음). 정예화
  해금 인프라 스킬(unlock이 `"정예화 1"`/`"정예화 2"`)이 있는 오퍼만 보유 오퍼 설정·방 상세에서
  **노정예/1정/2정**으로 낮출 수 있고, 낮추면 해당 단계 미만 해금 스킬이 스코어링·자동편성·방
  상세 표시 전부에서 제외된다(`withElite()`, `app/planner.tsx`). 성급별 일괄 버튼 제공
  (3성 이하는 2정이 없으므로 보유/해제만).

## 2. 스킬 적용 규칙 (데이터 구조가 정답)

- 강화/신규 구분은 `building_data`의 **buffChar 슬롯 구조**로 판정한다:
  - 같은 슬롯 안의 buffData 항목들 = **강화 단계** → 최종 단계만 적용
    (예: 피구리노 맞춤 재단 20% → 자유자재 25%, 합산 아님).
  - 서로 다른 슬롯 = **별개 스킬, 동시 적용** (예: 샤마르 재봉 α + 속삭임).
  - 이름 휴리스틱(α/β 접미사, 설명문 유사도)으로 추측하지 말 것.
- 수치 표기는 `+30%`와 `30% 상승` 두 형태. 정규식 필러가 숫자를 먹지 않도록 `[^%\d]` 계열을 쓴다
  ("50% 상승"이 "0% 상승"으로 잡히는 백트래킹 버그 전례 있음).
- 성장형 상한: "이후 시간당 2% 상승, **최대 30%까지**" → 상한값을 채택 (이네스).
  발전소도 동일 — "드론 상한 10당 +1% (최대 +25%)" → **+25% 고정** (이격그레이 점검 매뉴얼, 사용자 확정).
  **제조소도 동일 (사용자 확정 2026-07-20)** — "첫 시간 생산력 +15%, 그 이후 시간당 +2%,
  **최대 +25%**"(크루스·씬), "시간당 +1%, 최대 +25%"(팽·케오베), "시간당 생산력 +2%,
  **최대 +20%**"(아로마 정기 청소), "훈련실 레벨 1당 +10%, **최대 30%**"(Вий)는 전부 만렙
  기지 정상 운용 상한으로 계산한다. build-infra.py는 **순수 output이고 성장 문구(시간당·레벨
  1당)가 있을 때만** 상한을 적용한다 — 진영·계열 카운트 스킬의 "(최대 c%)"는 인원 상한이라
  건드리면 안 되므로(아몬드 블랙스틸 1명당 등) perFaction·perSkillTag 판정 이후에 건다.
- **자기 컨디션 낙차 페널티·게이트 (토터, 사용자 제보 2026-07)**: "생산력 +30%, 컨디션 낙차 4당
  -5%" + "낙차 12 이상일 경우 +10%"는 만컨디션(낙차 0) 최대치가 아니라 **대표 운용 낙차
  `DROP_ASSUMED=12`**(컨디션 9~12 구간, 12h 교대 평균)에서의 실효율로 보정한다 —
  흐려진 시야 30→15, 창밖 눈보라 +10 유지 = 실효 25%(버메일 재활용 조합 시 ~37%).
  40% 고정 표기(두 정점은 배타적: +30은 만컨디션, +10은 낙차≥12) 금지. build-infra.py가
  "컨디션 낙차 N당 생산력 -M%"·"컨디션 낙차 N 이상일 경우"를 파싱해 값에 반영.
- **정예화 하위단계 보존 (tiers, 스푸리아)**: 같은 buffChar 슬롯의 α(노정예)/β(2정)는
  최종(β)만 활성이되, 하위 단계를 최종 스킬의 `tiers`에 저장한다. 보유 오퍼 설정에서 정예화를
  낮추면 `withElite`가 잠긴 스킬을 **적용 가능한 최고 하위 tier로 대체**한다(없으면 제거).
  이전엔 최종만 저장해 노정예로 낮추면 스킬이 통째로 사라지는 버그가 있었다(스푸리아 기술 교류).

토큰 = 시설을 넘나드는 포인트. 생성자(+N)와 소비자(N점/개당 +효과)로 구성.

- **콜라보/특수 토큰 (COLLAB_TOKENS, 사용자 제보 2026-07-21)**: rules.json 카탈로그(본편 7종)엔
  없어 통째 미모델이던 세트를 `build-infra.py`의 `COLLAB_TOKENS`로 보강. 파서가 "N병당·N개
  늘어날 때마다·보유 N마다·N병 도달 시" 소비 문구와 "N개마다 전환"을 인식하도록 확장(엔진은
  기존 제네릭 토큰 흐름 그대로 — percent 소비자가 있으면 optimize가 자동 패키징). 표시명은
  `i18n.tsx` TOKEN_I18N(EN/JA)에 추가. Frost 문턱·Fuze cap·토가와 오라 등 비표준 소비는 근사.
- **미니 토큰 패키지는 planScore 비교로 이득일 때만 (사용자 제보 2026-07-21)**: 참가자 ≤4인
  토큰(아이룰루 등 콜라보 3~4인 세트, 다진영이라 폐쇄 팀 규칙을 비켜감)은 진영 세트와 같은
  규칙으로 **유/무 멱집합 planScore 비교** 후 이득일 때만 포함한다. 안 그러면 기대가치 시드가
  무조건 자리를 잡아 — 아이룰루가 테라 대륙 조사단을 무역소에 앉혀 **프로바이조 완성형
  (우요우+에벤홀츠+프로바이조)을 밀어내고** 레우스S·키린R이 제어센터 2석을 차지, 총점 −68
  손해인데도 강제됐다. 본편 대형 패키지(화식 등 5인+)는 항상 이득이라 종전대로 무조건 포함.
- **순금 생산 라인 차원 (투예·파죰카, 2026-07-21)**: 243 기지의 순금 제조소 = 2방(§1). "순금
  생산 라인 N개당 오더 수주 효율 +V%"는 (2÷N)×V를 무역 효율에 가산 (투예 5+15=20, 파죰카 2×5=10).
  build-infra.py가 절 앞부분으로 base 재파싱 후 가산. 키라라 '라인 +2'는 %없어 제외.
- **폐쇄 팀 규칙**: 참가자 전원이 단일 진영이고 2명 이상이면 기지 전체 패키지에서 제외한다.
  예외: 생성원이 전부 **숙소**에 있으면 허용(센시 마물 요리 — 경쟁 슬롯을 침범하지 않으므로).
- 숙소 기반 생성원(아이리스·체르니·비르투오사·센시)은 **숙소에 고정**되고 조 전환에 영향받지 않는다.
- 시드는 토큰 기대 가치 **20% 이상인 핵심 소비자만** 예약 배치한다. 약한 소비자(마르실 +5%급)는
  일반 경쟁 — 토큰 값은 점수에 자동 반영되므로 자리 낭비를 막는다.
- 시드된 오퍼는 해당 방에 **예약**된다(먼저 배정되는 방이 그리디로 채가지 못함).
- **죽은 전환 사슬의 공급원은 앉히지 않는다 (사용자 확정 2026-07-19)**: 전환으로만 토큰에
  기여하는 생성원(우요우: 화식→주술 결정)은 **전환자(지에윈)가 실제 배치돼 있을 때만** 자리를
  받는다. 무6성 로스터에서 기대가치 8짜리 사슬이 우요우를 무역소에 예약해 품질 조합
  (210 > 182.5)을 봉쇄하던 원인 — 픽스처 "샤마르+테킬라 품질 동반 (무6성 로스터)"로 고정.
- **자기 컨디션 조건 배타 생성 (시·링, 사용자 제보 2026-07)**: "컨디션 12 초과 시 속세의 화식
  +15, 12 이하 시 감지 정보 +10"처럼 자기 컨디션으로 갈리는 택1 생성은 **동시에 둘 다 계상하면
  안 된다**. 풀파워 A조는 컨디션이 높으므로 **"초과/이상" 브랜치만** 남긴다(파서가 토큰 앞
  ~30자에 "컨디션"+"미만/이하"가 있으면 그 토큰을 버림). 결과: 링→화식+15, 시→감지+10 (짝이
  합쳐 화식+감지 하나씩 — 이전엔 각자 둘 다 내서 이중계상).
- **모집 인원 초기분 제외 (멀베리·베이스라인)**: "모집 인원마다 +N (초기 모집 인원은 포함하지
  않음)" → 사무실 4슬롯 중 초기 2 제외 = **×2**(멀베리 화식 20, 베이스라인 공명 30. 사용자
  확정 2026-07). 제외 문구 없는 "모집 인원마다"는 종전대로 ×4.

## 4. 특수 역할 (무역소·제조소)

- **품질 조합 정배 (사용자 확정 2026-07, 재확정 2026-07-19)**: 미틈은 오더 효율 +30(응접실 레벨
  스케일)에 **고품질 귀금속 확률(quality)** 스킬이 있어 **샤마르(override 45)+테킬라(payout 25)
  조합과 같은 방**이 정배다. teamScore 모델이 이미 선호(샤마르+테킬라+미틈 205 vs 범용 대체 182.5):
  override는 인원당 45로 효율을 통일하고, quality 2장이 테킬라 payout 배율을 2.0까지 올린다.
  **샤마르+테킬라는 가능한 한 고품질 확률 요원(미틈·디아만테·카프카·바이비크 동급 대체)과
  묶는다** — 이를 구조적으로 보장하기 위해:
  ① `bestTeam` 쇼트리스트(단독 점수 상위 40)에 **팀 의존 역할군(override/payout/quality/
  percoworker)을 무조건 포함**한다 — 단독 점수로 저평가된 디아만테(15)가 잘려 그리디·감사가
  완성형을 영영 못 보던 문제.
  ② **품질 수익 조합 세트 후보안** — 다른 시드가 무역소를 선점·예약한 경우를 대비해
  [오버라이드+수익+품질] 3인 결집안을 만들어 planScore 비교로 채택 (쉐라그·피누스와 같은
  개별 후보 평가). **세트 정의는 rules.json `synergySets` 카탈로그(L2)**이고 엔진은 제네릭
  조립기만 가진다 — 새 팟 추가법은 PLANNER-RULES-DB §6.

- **효율 오버라이드(샤마르 속삭임)**: 같은 무역소 타인의 오더 효율을 전부 0으로 만들고 인당 +45%.
  → 방 효율 = 45 × 인원. 효율 요원과 섞지 말 것.
- **고품질 확률(재봉/수공예/감정 계열)**: 샤마르·카프카·디아만테·바이비크 — 전부 동급 대체 가능
  (α 소폭=10, β=15 등가).
- **품질 수익(테킬라 투자β)**: "순금 오더 납품 3초과" = 고품질 귀금속 오더. 확률 요원 수에 비례해
  증폭 → **샤마르+테킬라+확률 요원 1명**이 완성형.
- **위약 수익(프로바이조)**: 계약법이 납품 4미만을 위약 오더로 만들고 배상 +2. 고품질 확률과는
  **반시너지**(저품질 오더가 필요), 대신 **방 처리량(효율)에 비례해 증폭** →
  **우요우+에벤홀츠+프로바이조**가 완성형.
- **자동화(위디·유넥티스·윈드플릿·패신저)**: 방 내 오퍼 제공 생산력을 0으로 만들고
  **발전소 1기당** +15%/+10%/+5%/+5%. 단 "(시설 수량에 따라 제공하는 생산력 미포함)" —
  **시설 기반 생산력은 생존**: 퓨어스트림(무역소당 귀금속 +20%), 쏜즈 로드스타(무역소당 +3%) →
  **위디+유넥티스+퓨어스트림** 순금방 성립.
- **증폭형(와이후 협동의식·스노우상트 근면성실 β, 사용자 제보 2026-07-20)**: "해당 제조소/
  무역소 내의 **오퍼레이터가 제공한** 생산력/오더 수주 효율 **5%당 5%를 추가 제공, 최대 40%
  (스노우상트 35%)**" — 같은 방 다른 오퍼가 제공한 효율(**시설 수량 제공분 제외**)을 계단식
  배수로 되돌린다. `kind: "amplify"` + `amp:{per,add,cap}`. 엔진(teamScore)은 방의 오퍼 제공
  효율 합(`opProvided` = 파츠 efficiency 합, facilityEff·override·automation 제외)을 per로 나눠
  `min(cap, floor(opProvided/per)×add)`를 더한다. 제로아웃(override/automation)이 활성이면 제공
  효율이 0이라 증폭도 0. 팀 의존이라 단독 점수 0 → **TEAM_KINDS 쇼트리스트에 포함**해야
  자동편성이 증폭방을 짠다. build-infra.py 파서는 "오퍼레이터가 제공(하는/한) …%…당 …%
  **추가** … 최대 …%" 구조로만 감지 — 버메일 '늘린 용량당'·스와이어 '늘어난 상한만큼'·
  데겐블레허 'N**개**당'(용량 변환, capConv)은 걸리지 않는다. **정배**: 와이후+아로마(귀금속
  25+청소 20)+30%급 1명 = 75 제공 → +40 증폭 = **~115%**, 미즈키 표준화 3인 파티(~90%)보다
  강하다 — 표준화 파티가 시너지 없이 순금방에 앉던 제보를 해소.
- **자동화 · 인원수 스케일(스네구로치카)**: 같은 제로아웃 메커니즘이지만 발전소가 아니라
  **그 제조소에 실제 배치된 인원수 1명당** +10% — `kind: "automation_crew"`
  (`app/planner.tsx`의 `breakdown()`에서 `teamSize`로 스케일, `scripts/build-infra.py`
  파서는 "생산력이 전부 0이 되고 … 제조소 내의 오퍼레이터 1명당" 문구로 감지).
  일반 진영-카운트 정규식이 "제조소 **내의** 오퍼레이터 1명당"의 "내의"를 진영명으로 오인식하는
  버그가 있었음 — `내의`/`안의`/`내`는 진영 캡처 제외 목록에 있다.
- **제로아웃 오퍼 추천 원칙**: 방 내 모든 인원의 생산력을 0으로 만드는 오퍼(위 두 항목)를
  일반 +N% 생산력 오퍼와 같은 방에 추천하면 안 된다 — 그 +N%는 전부 0이 된다. 자동편성은
  실제 방 점수(제로아웃 반영)로만 비교하므로 이미 안전하지만, 새 오퍼 데이터를 다룰 때는
  "생산력이 전부 0이 되고" 문구가 있는데 `kind`가 `automation`/`automation_crew`/`override`가
  아니면(예: `output`으로 잘못 분류) 파서 버그다 — 제로아웃 오퍼를 쓰는 편성 자체가 예외적인
  케이스라는 것을 전제로 점검한다.
- **그레이 더 라이트닝베어러**: "다른 발전소에 배치된 작업 플랫폼이 없을 경우 발전소 +1개 간주".
  **작업 플랫폼 = 1성 로봇**이므로 로봇을 발전소에 안 넣으면 조건 충족 → 발전소에 배치하면
  발전소 4기로 계산되어 자동화 방이 140%가 된다.

### 용량 차원 — 오더 상한(무역소)·창고 용량(제조소) (2026-07-19 도입)

무역소의 **오더 상한**과 제조소의 **창고 용량**은 효율/생산력과 별개의 2차 축이다. 다수 오퍼가
효율/생산력과 **함께** 상한/용량을 ±N 부여하고, 일부 오퍼는 팀이 쌓은 상한/용량을 다시
효율/생산력으로 **변환**한다. 이 변환 계열이 핵심 시너지다 (사용자 제보 2026-07-19).

- **용량 부여** (`InfraSkill.cap`, 파서가 "창고 용량 상한/오더 상한 ±N"에서 추출): 실버애쉬 효율+20&상한+4,
  벌컨 생산력**-5**&용량+19, 베나 -20&용량+17, 데겐블레허 효율+25&상한**-6** 등. **음수도 반영** —
  파서 PCT가 `+N%`만 잡던 탓에 벌컨·베나·노시스의 음수 효율이 종전엔 misc/0으로 유실됐다.
- **용량→출력 변환기** (`InfraSkill.capConv`, `t`=lin/bundle/tier/diff):
  - `lin` 버메일 재활용(용량 1칸당 생산력 +2%) · 스와이어 디 엘리건트 위트 투자 유치(상한 1당 +4%)
  - `bundle` 데겐블레허 챔피언의 품격(제공 상한 5개당 +25%, 최대 100%)
  - `tier` 버블 큰 게 좋아 — **오퍼별 개별 용량 × 문턱식 (실측 역산 확정 2026-07-20)**:
    각 오퍼가 상승시킨 창고 용량이 16칸 이하면 **전량 1%/칸**, 16칸 초과면 **전량 3%/칸**
    (한계식 아님). 베나17→51 + 벌컨19→57 + 버블10→10 = **118%** − 페널티 25(베나 −20·벌컨
    −5) = **+93** = 게임 실측(방 +95 중 제조 오라 +2 제외분)과 정확 일치. 방 합산 한계식
    (46칸→106→81)·오퍼별 한계식(54→29)은 실측 불일치 — 세 가설 역산으로 판별했다.
    제어센터 앰비언트 상한(노시스)은 '자신이 상승시킨' 용량이 아니라 tier 계산에서 제외.
  - `diff` 제이 시장경제((기본상한+팀용량) 1당 +4%, 현재 오더 0 가정 — 유리 분기 관례)
- **엔진 (teamScore 2-패스)**: `capOp` = 오퍼 제공 용량 합(제어센터 앰비언트 포함, 0 미만 클램프),
  `capTotal` = `CAP_BASE`(§8) + capOp. 변환기가 capOp(lin/bundle/tier)·capTotal(diff)을 효율/생산력으로.
  방 상세엔 **'용량 변환'** 항목으로 표시된다. **베나+벌컨(용량↑) + 버블(변환)** 같은 조합이
  특정 룰 하드코딩 없이 **데이터에서 성립**한다 (verify-plan 픽스처 lock: 81 vs 50).
- **노시스 제어센터→무역소 (방 단위, 사용자 정정 2026-07-22)**: 정밀 계산 = "무역소 내 쉐라그
  1명당 오더 효율 **-15%**, 오더 상한 **+6**". **그 무역소에 실제 앉은 쉐라그 수**로만 스케일한다 —
  실버애쉬 1명 무역소면 -15%/+6, 쉐라그 0명 무역소면 0(유령 마이너스 없음), 제어센터의 노시스
  자신은 안 센다. 구현: `aurasOf`가 이 오라를 제조소 진영 오라(플레임테일)와 **같은 per-room 경로**로
  실어 보내고(`perProduct:{any:-15}` + `capPer:6`), `ambientFor`가 대상 무역소 팀의 쉐라그 수를
  직접 세 -15×수를 가산, `ambientCapFor`가 +6×수를 변환기(제이 시장경제·데겐블레허)에 먹인다.
  ⚠ 종전 버그(2026-07): 파서가 `perScope="base"`로 잡아 기지 전체 쉐라그 수(노시스 자기 포함)로
  스케일한 -45를 **모든 무역소에 flat 적용** → 실버애쉬/제이 무역소가 인겜 92% 대신 110%로,
  쉐라그 없는 무역소까지 마이너스가 새던 원인. verify-plan에 방-단위 회귀 2건으로 잠금.
- **쇼트리스트**: 벌컨(단독 -5%)·베나(-20%)는 단독 점수가 음수라 top-40에서 잘리므로,
  `cap`/`capConv` 스킬 보유자를 상시 포함군(TEAM_KINDS와 함께)에 넣어 변환기와 만날 기회를 준다.
- **한계**: base 상한값(CAP_BASE)은 근사(무역소 6). 제이 상인의 길·버메일↔버블 비중첩 규칙은
  근사 처리(둘 다 같은 방에 오면 합산 — 실제론 버블 우선). 니치 케이스라 허용, 추후 정밀화 대상.

## 5. 조건부 스킬 판정

- **조건부 가산 클래스 (`condBonus`/`perBase`, 사용자 제보 2026-07-21)**: "**기본 X% + 조건부
  Y%**" 복합 스킬. 종전 파서는 조건절로 **스킬 전체를 게이트**(기본치 소실)하거나 조건값을
  기본치로 오배치했다 — 르무엔(엑시아 없으면 0%)·뮤엘시스(50% vs 25%)·비나(글래스고 +10 누락)
  등 **11건**이 같은 원인(피드백 검증). `detect_cond_bonus`가 "추가[로] +Y%" 조건절을 분리해:
  - **roomFaction** `{value,faction}`: 같은 방에 그 진영원(본인 제외) 있으면 +value
    (비나+글래스고10·서퍼+블랙스틸20·Zima 이격+우르수스10·티폰+사미5).
  - **roomPartner** `{value,ids}`: 같은 방에 그 오퍼 있으면 +value (르무엔+엑시아25·발라크빈+티폰15).
    ⚠ **"엑시아"는 원본(char_103_angel)만** — 이격 엑시아 뉴커버넌트는 별개 id라 발동 안 함(정배).
  - **crossRoom** `{value,ids,room}`: 그 오퍼가 특정 방에 있으면 +value. roomPartner 게이트와
    같은 관례로 roomOf 없으면 낙관 통과(프라마닉스+실버애쉬레인프로스트@제어센터 10).
  - **basePartner**: "X가 **기반시설/숙소에** 있으면 +Y" → 기존 `basePartnerBonus` 재사용
    (Bellone+비질10·산크타+피아메타10 — 언더플로우와 동일. "있을 시"도 잡도록 확장).
  - **perFacBase** (`perBase`): 진영 카운트 + **별도 flat 기본치**. 뮤엘시스 "회복 +10% +
    라인 랩 1명당 +3%(최대 5=15)" → value=3(per-member)·perBase=10·perCap=15, 엔진이 perBase를
    카운트분과 별도로 가산 (종전엔 기본 10을 per-member로 오인해 10/명·상한 50). ⚠ **모건**
    (per-Glasgow + 시즈 보너스 복합)은 자기-카운트 미해결이라 이번 자동수정 **제외** — 후속.
  파서는 순수 조건부(기본치 없음: 알라나·불피스폴리아)엔 손대지 않고(등가), "추가" 문구 없거나
  유형 미상(훈련 마스터리·토큰·레벨성장 52건)은 원문 유지. 부분문자열 사전필터로 감지 O(1)에 근접.
- **진영 동반 게이트**: "X 오퍼레이터와 함께 배치 시" → 같은 방에 그 진영 오퍼(본인 제외)가 실제로
  있어야 발동 (호시구마 더 브리처 동료애: 용문근위국).
- **진영 카운트**: "X 오퍼레이터 1명당 +v% (최대 c%)" → 실제 배치 인원 기준, 상한 적용
  (바르카리스: 미노스). 범위는 "제어 센터 내"면 방, 아니면 기지 전체.
  - **"자신을 제외한" self 제외 (`perExclSelf`, 뮤엘시스)**: 본인이 그 진영이면 카운트에서
    자신을 뺀다 — build-infra.py가 "자신을 제외" 문구로 감지, 엔진이 count−1. 안 빼면 뮤엘시스
    단독이 10이 아니라 13(자기 +3)으로 뜬다.
  - **인프라 진영은 이탈 이격을 제외 (`INFRA_FACTION_REMOVE`, 사용자 확인 2026-07-21)**:
    게임 RIIC의 진영 그룹은 character_table `groupId`와 달리 **스토리상 그 진영을 탈퇴한 이격
    오퍼를 뺀다**. 라인 랩은 groupId=rhine 12명 중 인게임 명단(10명)에서 이격 2명
    (**사일런스 더 패러디그매틱·아스트젠 더 라이트체이서**, 둘 다 로도스 이적)만 빠졌다 —
    인게임 크로스체크 확인. buffId가 리치텍스트 그룹 참조라 명단 파일이 없어 build-infra.py에
    명시. **인프라 전용** (operators.json 표시/프로필 진영은 groupId 그대로) — 새 이탈 이격이
    나오면 (id, 뺄 진영) 추가. ⚠ 컨셉덱 시너지엔 아직 미반영 (필요 시 확장).
  - **진영명은 알려진 진영으로 한정 매칭**(FACTION_NAMES 최장 우선) — 공백 포함 진영명(**라인 랩**)도
    잡고, "제조소 **내의** 오퍼레이터"·"숙소 **내**" 같은 방 조사구를 진영으로 오인하지 않는다
    (내스티·아몬드·뮤엘시스가 flat/미파싱이던 버그 수정, 2026-07).
  - **개수 상한 "(최대 N명)"** (오퍼레이터와 "1명당" 사이): 퍼센트 상한 등가로 변환
    `perCap = value × N` (내스티 3%×5명=15%). "최대 c%"(퍼센트 상한)와 구분.
- **제어센터에서 무역소 내 진영 카운트 오라 (야하타 우미리·델핀·노시스)**: "무역소 내 <진영> 1명당
  오더 수주 효율 ±v%"는 `misc`가 아니라 **`ctrl_trade` 오라**(perFaction). **대상 무역소에 앉은 진영원
  수로 방별 적용** — 진영 0명이면 0(야하타 시라쿠사 +5·델핀 글래스고 +10·노시스 쉐라그 -15 전부
  동일 규칙, 위 노시스 항목 참조). ⚠ 종전엔 `perScope="base"`라 기지 전체 진영 수로 스케일해 flat
  적용되던 버그 — `aurasOf`가 제조소 진영 오라와 같은 per-room 경로로 실어 방별로 카운트하도록
  정정(사용자 제보 2026-07-22).
- **작업 플랫폼 발전소 게이트 (푸딩 오버클럭)**: "작업 플랫폼(1성 로봇) N대 이상 발전소 배치 시
  제조소 +2%"는 자동편성이 로봇을 발전소에 넣지 않아 **사실상 미충족** → `gatePlatforms`로
  표시하고 플래너가 오라를 계상하지 않는다(스킬은 방 상세에 표시하되 효과 0). 그레이(§4)가
  로봇을 발전소에서 빼려는 것과 상충하는 조건이기도 함.
- **동방 스킬 카운트**: "제조소 내 금속 공예류 스킬 1개당 +5%" → 팀원의 활성 스킬 이름 매칭
  (브라이오피타).
- **시설 카운트**: "각각의 무역소가 … +3%" → 무역소 2 / 발전소 3(또는 4) / 제조소 4 배수.
- **동료 카운트**: "자신을 제외한 작업 중인 오퍼레이터 1명당 +10%" (키치세이).
- **단독 조건**: "자신만 업무 중일 때" → 혼자일 때만 (아르모니). **단서 공유 조건**: "단서 공유
  상태에서" → 액면가로 인정.
- **레벨 성장형은 만렙 기지 기준 상한 채택** (이네스·이격그레이 원칙의 확장, 사용자 확인):
  - "응접실 레벨 1레벨당 추가 +5%, 최대 40% 제공" → 응접실 Lv3 = **상한값 채택** (비질 40, 미틈 30).
  - "**모든 숙소의 레벨 1당** +N%" → 숙소 4 × Lv5 = **×20** (아르케토 40, 틴맨 45,
    나란투야 20(순금), 필라에 10). 파서는 이 절을 떼고 기본치를 파싱한 뒤 ×20을 더한다.
  - "정예 오퍼레이터가 배치된 시설 1개당 +2% (최대 10개)" → 풀 기지는 시설 10개를 쉽게
    채우므로 상한 채택 (만트라 45).
- **응접실 레어도·정예화 기본 단서속도 (사용자 지적 2026-07 · Terra Wiki)**: 단서 수집
  속도는 RIIC 스킬 수치와 **별개로** 오퍼 레어도·정예화 기본치가 가산된다 —
  **6성 +10 / 5성 +9 / 4성 +7 / 3성↓ +5**, **정예화 E1 +8 · E2 +16** (정예화 미지정은
  그 레어도의 최대 승급 가정: 6·5·4성 E2, 3성 E1, 2성↓ E0). RIIC 스킬 없는 2정 6성도
  **+26%**, 2정 5성 +25%(=12F: 기본 5 + 스킬 20). 응접실 2인 배치라 두 오퍼 값을 합산한다.
  구현은 `planner.tsx`의 `clueBase()` — `OpBreakdown.clueBase`로 분리 계산해 점수에 넣고
  카드에는 **'레어도 기본 N성·정예화'**로 스킬 효율과 따로 표기한다(사용자 요청).
- **per-faction 방/기지 범위 (사용자 지적 2026-07)**: "함께 배치된 X 1명당"·"해당 무역소/
  제조소/발전소 내 X 1명당"은 **방 범위**(같은 방 진영 동료만; `perScope="room"`), "기반시설 내
  배치된 X 1명당"은 **기지 전체**(`base`)다. build-infra.py가 방 범위 문구를 base로 오판정해

  `find_partners`가 매치 지점이 더 긴 진영명으로 시작하면 무시한다 (사용자 제보로 발견:
  실버애쉬 이격·노시스가 응접실 전용 쉐라를 제어센터에 끌고 오던 버그).
- **공사용 로봇 세트 (미니멀리스트)**: "시설마다 레벨당 공사용 로봇 +1대 (최대 64대)" +
  "로봇 8대당 생산력 +5%" — 만렙 기지 시설 레벨 합이 정확히 64라 상한을 채우므로
  결합 가치 **+40%** (64÷8×5). 파서가 형제 스킬의 로봇 상한과 소비 배율을 결합한다.

## 6. 제어 센터

- 오라 우선순위(사용자 확정): **제조소 생산력 > 무역소 오더 효율 > 인맥 레퍼런스 > 단서 수집**.
  가중치: mfg ×10, trade ×2, hire ×0.6, clue ×0.2.
- "동종 효과 중 가장 높은 수치만 적용" — 같은 종류 오라는 팀 내 최대값 하나만.
- 쉐이 3인(링·시·총웨)이 화식 생성으로 고정, 남는 2자리를 오라 요원으로 채움.
- **훈련 속도 제어 스킬은 생산가치 0 (사용자 확인 2026-07-22)**: "제어 센터 배치 시, 훈련실에서
  스킬 마스터리 훈련 중인 오퍼레이터의 스킬 훈련 속도 +N%"(이격 블레이즈 단금지교·아스카론·
  호시구마 더 브리처)는 생산 자원을 내지 않아 정상 운용 생산가치 0이다(훈련실 훈련속도 하드
  스킵과 같은 원칙). build-infra.py CONTROL 분기가 "훈련 속도"를 만나면 `misc, 0`으로 잡는다 —
  종전엔 generic PCT 폴백이 +5%를 misc 효율로 잡아 제어센터 자리 점수에 새어들어, 강한 오라
  요원이 없는 로스터에서 **이격 블레이즈가 제어실에 배치**되던 원인. 표시는 operators.json의
  description 기반이라 value 0이어도 도감·방 상세엔 원문(+N%)이 그대로 나온다.

## 7. 대체 추천

- **슬롯별 교체 시뮬레이션**: 그 자리에 후보를 넣었을 때의 방 점수로 순위. 동점이면 저성급 우선
  (육성 저렴 — 그라벨 원칙).
- **시너지 코어는 대체 불가**: 활성 토큰의 생성자/소비자, 오버라이드·수익 역할, 인원 카운터 대상
  (쉐이 진영) → "대체 불가 · 시너지 코어" 표시. 디아만테처럼 역할 호환 자리만 대체 후보 제공.

## 9. 미래시(미실장) 오퍼 (2026-07 도입)

- `build-infra.py`가 미실장 오퍼(operators.json `unreleased: true`)의 buffChar를
  **CN building_data에서 폴백**으로 읽고, 설명·이름을 한국어화한 뒤(양 서버 공존 buffId를
  조인한 CN→KR 자동 사전 + `scripts/cn-translations.json` 비공식 AI 번역) 기존 한국어
  파서에 그대로 태운다 — 번역이 파서 정규식(공식 문구)을 따라야 수치가 잡히므로, 번역
  집필 시 "오더 수주 효율/드론 회복/부산물 산출 확률/컨디션 회복/만약 X가 Y에 배치되어
  있다면" 같은 공식 표현을 지킬 것. 미번역이 남으면 UNTRANSLATED 경고 + 수치 0.
- infra.json 오퍼에 `unreleased: true` 플래그가 실리고, **플래너는 헤더 '미래시 데이터
  포함' 토글이 켜져야** 로스터·보유 설정·자동편성에 포함한다 (보유 설정엔 미실장 배지 표시).
  토글을 바꿔도 현재 편성은 유지 — 다음 자동편성부터 반영 (설정 불변 원칙).
- 미실장 파트너·교차방 조건은 한글 통칭으로 매칭한다 — 표시명은 영문 코드네임이라
  `name_to_id`에 미실장 오퍼의 aliases(타마미츠네 오키드·클로저 등)를 등록하고, 자기
  언급은 id로 거른다(클로저 '특별 오더'). partners와 roomPartner 이중 게이트 금지 —
  roomPartner id는 partners에서 제외 (레토는 '굼'이 1글자라 우연히 회피했던 부분의 명시화).
- EN/JA: 미실장 buffId는 로케일 테이블에 없으므로 `build-i18n.py`가 infra.json의 한국어
  텍스트를 수확 사전+수동(ko→로케일)으로 번역해 extra-i18n buffs에 채운다.
- KR 정식 출시되면 KR building_data가 우선하므로 자동으로 공식 데이터·번역으로 대체된다.

## 10. 신규 오퍼레이터 추가 시 점검 목록

1. `python3 scripts/build-infra.py <gamedata>` 재실행 후 신규 오퍼의 파싱 결과 확인
   (kind / value / product / tokenGen / tokenUse / 조건 필드).
2. 새 토큰 시스템이면 `app/data/rules.json`의 `tokens` 목록에 추가하고 생성·소비·전환
   문구가 잡히는지 확인.
3. 파서가 오분류한 스킬은 우선 `rules.json`의 `skillOverrides`(buffId→patch)로 교정하고,
   "…당", "…와 함께", "전부 0이 되고", "간주" 같은 **새 조건 문구 유형**은 이 문서에 규칙을
   추가하고 파서·플래너에 반영 (정식 지원되면 override 행 삭제 — 미적용 행은 재생성 시 WARNING).
4. 플래너 로직 검증은 `node scripts/verify-plan.mjs` — 픽스처(검증된 정배·절대룰) 회귀 검사.
   엔진 리팩토링은 `--snapshot`으로 기준을 뜨고 수정 후 `--compare`로 무변화를 증명한다.
5. 확정된 새 규칙은 이 문서와 도움말 모달(`planner.tsx`의 HelpModal)에 함께 갱신하고,
   편성 정배로 확정된 조합은 `rules.json`의 `fixtures`에 케이스로 축적한다.
